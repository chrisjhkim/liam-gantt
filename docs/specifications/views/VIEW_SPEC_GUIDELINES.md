# View 명세서 작성 가이드라인
> 기획자 관점의 화면 명세서 작성 방법 및 규칙

## 📋 개요
이 문서는 Liam Gantt 프로젝트의 View 명세서를 작성할 때 따라야 할 가이드라인입니다. **기획자의 관점**에서 화면의 기능과 동작을 상세히 기술하되, 기술적 구현 사항은 배제하고 순수한 기능 명세에 집중합니다.

## 🎯 작성 목적
1. **기능 명확화**: 화면에서 사용자가 할 수 있는 모든 작업 정의
2. **개발 가이드**: 개발자가 구현해야 할 기능과 동작 명시
3. **일관성 유지**: 모든 화면의 UX 패턴 표준화
4. **AI 활용**: Claude 등 AI가 정확한 기능 구현 가능하도록 상세 기술

## 🎯 작성 원칙

### 1. 기획자 관점 유지
- ✅ **기능 중심**: 사용자가 무엇을 할 수 있는가
- ✅ **동작 중심**: 버튼을 누르면 어떤 일이 일어나는가
- ✅ **결과 중심**: 화면에 무엇이 표시되는가
- ❌ **기술 구현**: HTML, CSS, JavaScript 코드
- ❌ **개발 용어**: Controller, DTO, API 엔드포인트 등

### 2. 구체적이고 상세한 기술
모든 UI 요소와 동작을 **빠짐없이** 기술해야 합니다.

#### 올바른 예시
```markdown
### 2-1. 새 프로젝트 버튼
- **위치**: 화면 우상단
- **버튼 텍스트**: "새 프로젝트 만들기"
- **스타일**: 파란색 배경, 흰색 글자
- **기능**: 클릭 시 새 프로젝트 생성 페이지(`PAGE-PROJECT-002`)로 이동
- **표시 조건**: 로그인한 사용자에게만 표시
```

#### 잘못된 예시
```markdown
### 버튼들
- 새 프로젝트 버튼이 있음
- 클릭하면 이동함
```

### 3. 페이지 ID 시스템 활용
모든 페이지와 화면은 고유한 ID를 가져야 합니다.

#### 네이밍 규칙
- **형식**: `PAGE-[SECTION]-[NUMBER]`
- **예시**:
  - `PAGE-HOME-001`: 홈 페이지
  - `PAGE-PROJECT-001`: 프로젝트 목록
  - `PAGE-TASK-002`: 태스크 상세

#### 상호 참조 방법
```markdown
- **기능**: 클릭 시 프로젝트 상세 페이지(`PAGE-PROJECT-003`)로 이동
```

## 📖 문서 구조

### 1. 문서 헤더
```markdown
# [Section] Views Specification
> [섹션] 관련 화면 기획서

## 📋 Overview
[해당 섹션 화면들의 전반적인 설명]

## 📄 페이지 정의 목록
이 문서에서 정의하는 페이지들의 고유 ID와 설명입니다.

| 페이지 ID | 페이지명 | 설명 |
|-----------|----------|------|
| `PAGE-XXX-001` | 페이지명 | 설명 |
```

### 2. 개별 페이지 섹션
```markdown
## 페이지명 {#PAGE-XXX-001}
> 페이지에 대한 한 줄 설명

### 화면 구성 개요
[페이지의 전반적인 목적과 역할 설명]

### 1. [주요 섹션명]
#### 1-1. [세부 구성요소]
- **구성요소명**: 상세 설명
- **위치**: 화면상 위치
- **기능**: 어떤 동작을 하는지
- **표시 조건**: 언제 보이는지
```

### 3. 공통 요소 표준화

#### 버튼 기술 방법
```markdown
#### X-X. 버튼명
- **위치**: 화면상 구체적인 위치
- **버튼 텍스트**: "정확한 텍스트"
- **스타일**: 색상, 크기 등 시각적 특징
- **기능**: 클릭 시 발생하는 동작 (페이지 이동, 팝업 등)
- **표시 조건**: 언제 활성화/비활성화되는지
- **권한**: 필요한 경우 접근 권한 명시
```

#### 입력 필드 기술 방법
```markdown
#### X-X. 필드명 입력
- **라벨**: "표시되는 라벨명"
- **입력 타입**: 텍스트/숫자/날짜/드롭다운 등
- **필수 여부**: 필수 입력 또는 선택 입력
- **최대 글자수**: 제한이 있는 경우
- **플레이스홀더**: "안내 텍스트"
- **유효성 검사**: 어떤 조건을 만족해야 하는지
- **기본값**: 초기에 설정되는 값
```

#### 목록/테이블 기술 방법
```markdown
#### X-X. 목록/테이블명
##### 테이블 헤더
- **컬럼명1** (20% 너비): 표시 내용 설명
- **컬럼명2** (30% 너비): 표시 내용 설명

##### 각 행 구성
- **데이터1**: 어떤 데이터가 어떻게 표시되는지
- **데이터2**: 표시 형식 (날짜, 숫자, 배지 등)
- **액션 버튼들**: 각 행에서 수행할 수 있는 작업들
```

## 🎨 UI 요소 기술 표준

### 1. 상태 표시
```markdown
- **배지 형태**로 상태 표시:
  - 진행중: 파란색 배지
  - 완료: 초록색 배지
  - 중단: 빨간색 배지
```

### 2. 진행률 표시
```markdown
- **진행률 바**: 가로 막대 형태
  - 0-30%: 빨간색
  - 31-70%: 노란색
  - 71-100%: 초록색
- **퍼센트 숫자**: 진행률 바 옆에 "85%" 형태로 표시
```

### 3. 날짜 및 시간 표시
```markdown
- **날짜 형식**: "MM/DD" 또는 "YYYY-MM-DD" 형태
- **시간 형식**: "HH:mm" 형태
- **상대 시간**: "3일 전", "2시간 후" 등
```

### 4. 빈 상태 화면
```markdown
### X. 빈 상태 화면
데이터가 없을 때 표시되는 화면입니다.

#### X-1. 표시 내용
- **아이콘**: 큰 빈 상태 아이콘
- **메시지**: "등록된 항목이 없습니다"
- **안내 텍스트**: "첫 번째 항목을 추가하여 시작하세요"

#### X-2. 액션 버튼
- **항목 추가하기** (파란색): 새 항목 생성 페이지로 이동
```

## 🔄 화면 간 이동 흐름

### 1. 이동 흐름 섹션 구성
모든 명세서 끝에 다음 섹션을 포함해야 합니다:

```markdown
## 화면 간 이동 흐름

### 주요 이동 경로
1. **시작점 → 도착점**: 어떤 액션으로 이동하는지
2. **페이지A → 페이지B**: 구체적인 버튼명이나 액션

### 저장 후 이동 규칙
- **작업 완료**: 어디로 이동하는지
- **작업 취소**: 어디로 돌아가는지

### 오류 발생 시
- **유효성 검사 실패**: 어떻게 처리되는지
- **서버 오류**: 사용자에게 어떤 메시지가 표시되는지
```

## ⚠️ 피해야 할 표현들

### 1. 기술적 용어 사용 금지
❌ **잘못된 표현**:
- "Controller에서 Model에 데이터를 전달"
- "AJAX로 비동기 처리"
- "JavaScript로 DOM 조작"
- "Bootstrap 클래스 적용"

✅ **올바른 표현**:
- "버튼 클릭 시 데이터가 서버에 저장됨"
- "페이지 새로고침 없이 결과가 표시됨"
- "화면 요소가 동적으로 변경됨"
- "반응형으로 화면 크기에 맞춰 조정됨"

### 2. 모호한 표현 금지
❌ **모호한 표현**:
- "적절히 배치됨"
- "사용자 친화적으로 표시"
- "효과적으로 처리"

✅ **구체적인 표현**:
- "화면 중앙에 세로로 정렬됨"
- "큰 글씨와 명확한 아이콘으로 표시"
- "3초 이내에 결과가 화면에 나타남"

## 🔍 검토 체크리스트

### 문서 완성도 확인
- [ ] 모든 UI 요소가 기술되었는가?
- [ ] 모든 버튼의 동작이 명시되었는가?
- [ ] 페이지 ID가 일관성 있게 적용되었는가?
- [ ] 화면 간 이동 흐름이 명확한가?

### 기술적 내용 제거 확인
- [ ] HTML/CSS/JavaScript 코드가 포함되지 않았는가?
- [ ] 개발 용어(Controller, API 등)가 사용되지 않았는가?
- [ ] 구현 방법 대신 기능과 동작에 집중했는가?

### 명확성 확인
- [ ] 기획자가 아닌 사람도 화면을 상상할 수 있는가?
- [ ] 각 요소의 위치와 동작이 구체적으로 기술되었는가?
- [ ] 예외 상황과 오류 처리가 포함되었는가?
